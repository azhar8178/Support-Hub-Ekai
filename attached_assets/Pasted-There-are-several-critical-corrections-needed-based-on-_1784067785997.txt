There are several critical corrections needed based on Ekai's 
deployment architecture. Ekai's platform runs inside customer 
private cloud environments (AWS VPCs, Azure VNets, GCP VPCs). 
The support portal at support.ekai.ai CANNOT reach into customer 
networks. Customer deployments push outbound to us — we never 
pull inbound from them.

- Default mode for all new environment registrations = CLIENT PUSH

  URLs (dev/staging environments explicitly)
- On the Register Environment form, add a field:
    Heartbeat Mode:
    (*) Client Push — deployment sends heartbeats to this portal
                      [RECOMMENDED — works in private networks]
    ( ) Hub Poll    — portal fetches from deployment URL
                      [Only for publicly accessible environments]
- Add a warning banner on Hub Poll selection:
    "⚠ Hub polling requires the deployment URL to be publicly 
    reachable from the internet. Not suitable for production 
    deployments in private VPCs."

CORRECTION 4 — HEALTH AGENT SCRIPT

Create /agent/health-agent.py — a standalone Python script 
that runs inside the customer's deployment and sends 
heartbeats to the portal. This is the CLIENT PUSH agent.

Requirements:
- Pure Python — only stdlib + requests + psutil
- Works on Python 3.8+
- Zero framework dependencies
- Runs as a long-running process OR as a one-shot 
  (called by external scheduler)

Configuration via environment variables:
  FLEET_HUB_URL      - https://support.ekai.ai
  FLEET_API_KEY      - the token from registration
  FLEET_INTERVAL     - push interval seconds (default: 300)
  FLEET_SERVICES     - JSON array of services to probe
  FLEET_CLOUD        - aws | azure | gcp | other
  FLEET_REGION       - e.g. eu-west-1
  FLEET_RUNTIME      - ecs | eks | aks | gke | docker | 
                       k8s | vm | other
  FLEET_VERSION      - current Ekai deployment version

FLEET_SERVICES format:
[
  {
    "name": "ekai-api",
    "type": "api",
    "url": "http://localhost:8080/health",
    "timeout": 5
  },
  {
    "name": "ekai-db",
    "type": "database",
    "url": "http://db-host:5432",
    "timeout": 3
  }
]

Service health logic:
- HTTP/HTTPS URL: GET request
    200-399 under 500ms  = healthy
    200-399 500-2000ms   = degraded  
    non-200 or >2000ms   = degraded
    timeout / refused    = down
- TCP host:port (no http): socket connect
    connects under timeout = healthy
    timeout / refused      = down
- psutil for CPU/memory if available, null if not

Overall status:
  any service = down     → down
  any service = degraded → degraded
  all healthy            → healthy

Script structure:
1. Load and validate all env vars on startup
   (exit with clear error if FLEET_HUB_URL or 
   FLEET_API_KEY missing)
2. Log startup: "Ekai Fleet Agent starting. 
   Monitoring X services. Push interval: Ys"
3. Loop:
   a. Probe all services
   b. Collect system metrics
   c. Build payload
   d. POST to FLEET_HUB_URL/api/fleet/heartbeat
      with X-Fleet-API-Key header
   e. Log: timestamp, status, HTTP response, 
      next push in Ys
   f. On HTTP error: log warning, do not crash
   g. Sleep FLEET_INTERVAL seconds
4. Handle SIGTERM gracefully (log shutdown, exit 0)

Also create /agent/Dockerfile:
FROM python:3.11-slim
RUN pip install --no-cache-dir requests psutil
WORKDIR /agent
COPY health-agent.py .
CMD ["python", "-u", "health-agent.py"]

Also create /agent/README.md with deployment instructions 
for each platform. The README must include copy-paste 
commands for:

1. Docker (universal):
   docker run -d --name ekai-fleet-agent \
     -e FLEET_HUB_URL=https://support.ekai.ai \
     -e FLEET_API_KEY=ek_fleet_xxx \
     -e FLEET_CLOUD=aws \
     -e FLEET_REGION=eu-west-1 \
     -e FLEET_RUNTIME=ecs \
     -e FLEET_VERSION=1.0.0 \
     -e FLEET_SERVICES='[{"name":"ekai-api","type":"api","url":"http://ekai-api:8080/health","timeout":5}]' \
     --restart unless-stopped \
     ekai/fleet-agent:latest

2. AWS ECS (add as sidecar container in task definition):
   JSON snippet showing the container definition with 
   all env vars as secrets from Secrets Manager

3. Azure Container Instances:
   az container create command with env vars

4. GCP Cloud Run Jobs:
   gcloud run jobs create command with env vars

5. Kubernetes (any cloud) CronJob:
   Full YAML manifest with env vars from a Secret

6. Linux VM / bare metal:
   crontab entry calling docker run
